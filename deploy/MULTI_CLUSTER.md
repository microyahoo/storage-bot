# Multi-Cluster Kubeconfig Operations Guide

How to collect kubeconfigs from N managed Ceph clusters and feed them to storage-bot.

## Step 1: Generate a bot-specific kubeconfig per cluster

Don't ship the cluster-admin kubeconfig — the bot only needs read access to one
namespace. On each managed cluster (assumes `rook-ceph` namespace):

```bash
CLUSTER=cluster-01
NS=rook-ceph

# 1. ServiceAccount + read-only Role
kubectl -n $NS create serviceaccount storage-bot
kubectl create clusterrole storage-bot-reader \
  --verb=get,list,create \
  --resource=pods,pods/exec,pods/log
kubectl create clusterrolebinding storage-bot-$CLUSTER \
  --clusterrole=storage-bot-reader \
  --serviceaccount=$NS:storage-bot

# 2. Pull the long-lived token (k8s 1.24+ needs an explicit Secret)
cat <<EOF | kubectl -n $NS apply -f -
apiVersion: v1
kind: Secret
metadata:
  name: storage-bot-token
  annotations:
    kubernetes.io/service-account.name: storage-bot
type: kubernetes.io/service-account-token
EOF

TOKEN=$(kubectl -n $NS get secret storage-bot-token -o jsonpath='{.data.token}' | base64 -d)
CA=$(kubectl -n $NS get secret storage-bot-token -o jsonpath='{.data.ca\.crt}')

# 3. Write a minimal kubeconfig. server: should be the apiserver address
#    that the bot pod can reach — see Step 2.
APISERVER=https://10.0.1.10:6443   # adjust per cluster

cat > ${CLUSTER}.kubeconfig <<EOF
apiVersion: v1
kind: Config
clusters:
- name: $CLUSTER
  cluster:
    server: $APISERVER
    certificate-authority-data: $CA
contexts:
- name: $CLUSTER
  context:
    cluster: $CLUSTER
    namespace: $NS
    user: storage-bot
current-context: $CLUSTER
users:
- name: storage-bot
  user:
    token: $TOKEN
EOF
```

Result: a `cluster-01.kubeconfig` file with read-only access to one namespace.

## Step 2: Pick the right apiserver address

The kubeconfig `server:` field must point to an address the **bot pod** can reach.
Common pitfalls and fixes:

| Cluster setup | Default `server:` | What to do |
|---|---|---|
| kubeadm single-master | `https://127.0.0.1:6443` | Replace with master node IP — that IP must already be in the apiserver cert SANs |
| Internal DNS | `https://kube-api.local:6443` | Replace with reachable IP, or run the bot inside the same DNS zone |
| HA via VIP | `https://vip:6443` | Use the VIP, ensure firewall allows the bot's source IP |
| Cloud LB | `https://lb.example.com:6443` | Often works as-is |

You have two options to apply the new address:

**Option A — bake it into the kubeconfig file** when generating (the example above).
Then `server_override` in `config.yaml` can stay empty.

**Option B — leave the original kubeconfig untouched, override at runtime:**
```yaml
clusters:
  cluster-01:
    kubeconfig: /etc/storage-bot/kubeconfigs/cluster-01
    server_override: "https://10.0.1.10:6443"
```

Option B is better when the kubeconfigs are auto-generated by some tool and you
don't want to mutate them.

### Certificate SAN mismatch

If `server_override` points to an IP / hostname not in the apiserver cert SANs,
you'll see `x509: cannot validate certificate for ... because it doesn't contain
any IP SANs`. Two fixes:

1. **Right way**: regenerate the apiserver cert with the new SAN (kubeadm:
   `--apiserver-cert-extra-sans`).
2. **Quick way**: set `insecure_skip_tls_verify: true`. Acceptable for an
   internal management tool on a trusted network; not for prod auditing.

## Step 3: Mount kubeconfigs into the bot pod

Put each kubeconfig in a ConfigMap (or Secret if you treat the token as sensitive):

```bash
# Option A: one ConfigMap with multiple keys
kubectl -n storage-bot create configmap storage-bot-kubeconfigs \
  --from-file=cluster-01=cluster-01.kubeconfig \
  --from-file=cluster-02=cluster-02.kubeconfig \
  --from-file=cluster-03=cluster-03.kubeconfig
```

The Deployment in `deploy/manifests.yaml` mounts this at
`/etc/storage-bot/kubeconfigs/`. Each cluster's kubeconfig appears as a file
named after its cluster — the `kubeconfig:` field in `config.yaml` should point
to that path.

## Step 4: Adding a new cluster (no restart needed)

1. Generate kubeconfig on the new cluster (Step 1).
2. Update the ConfigMap:
   ```bash
   kubectl -n storage-bot create configmap storage-bot-kubeconfigs \
     --from-file=cluster-01=cluster-01.kubeconfig \
     --from-file=cluster-02=cluster-02.kubeconfig \
     --from-file=cluster-21=cluster-21.kubeconfig \
     --dry-run=client -o yaml | kubectl apply -f -
   ```
3. Update the `storage-bot-config` ConfigMap to add the new cluster entry.
4. The file watcher detects the change (within 30s) and reloads. No pod restart.
   If you want immediate reload: `kubectl -n storage-bot exec deploy/storage-bot -- kill -HUP 1`.

## Step 5: Verify connectivity

```bash
kubectl -n storage-bot exec deploy/storage-bot -- \
  curl -sk --cacert /dev/null https://10.0.1.10:6443/healthz
```

If that returns `ok`, the bot pod can reach the cluster's apiserver. If it
hangs, your network is the problem (security group, route, VPN).
